# Expense-Agent：AI 辅助报销审核系统
## AI产品经理设计文档 v1.0

---

## 一、产品定位

**产品名称**：Expense-Agent 智能报销审核平台
**核心价值主张**：用 AI 替代报销流程中的重复性合规判断，将财务专员从"票据警察"解放为"财务顾问"

> 一句话定位：**让每一张发票都有政策依据，让每一次退回都有标准话术。**

---

## 二、业务背景

### 企业画像

| 维度 | 数据 |
|---|---|
| 员工规模 | 约 2,000 人 |
| 出差人员占比 | 约 70%（业务与交付人员） |
| 月均报销单数 | 约 5,000 份 |
| 月均单据明细 | 超过 18,000 条 |
| 财务配置 | 每个办公室 1-2 名财务专员 |
| 分公司覆盖 | 上海（总部）、北京、深圳、成都 |

### 现有报销流程

```
员工提交报销
  ├─ 手工录入费用明细
  ├─ 拍照上传票据图片
  ├─ 打印报销单
  └─ 放入报销箱
        ↓
财务初审（人工，10-15分钟/单）
  ├─ 核对系统记录
  ├─ 检查票据类型
  └─ 异常退回处理（退回率 20%）
        ↓
复核与打款
  ├─ 月底集中复核
  ├─ 生成付款汇总
  └─ 批量转账
```

## 三、核心痛点分析

| 痛点类别 | 具体问题 | 业务影响 | 影响烈度 |
|---|---|---|---|
| **审核效率低** | 每张报销单平均审核 10-15 分钟，月底集中爆发 | 财务人力紧张，审核周期拉长 | ⭐⭐⭐⭐⭐ |
| **错误率高** | 退回率高达 20%：抬头错误、金额计算误差、重复报销 | 增加沟通成本，员工体验差 | ⭐⭐⭐⭐ |
| **合规风险** | 票据内容模糊、手写金额、日期不符 | 财务风险，影响内审 | ⭐⭐⭐⭐⭐ |
| **数据分散** | 电子系统与纸质并行，附件缺失 | 难以形成完整审核数据集 | ⭐⭐⭐ |
| **员工体验差** | 报销周期长，无法追踪进度 | 满意度下降，频繁催促财务 | ⭐⭐⭐ |

### 根因分析

```
退回率 20% 的构成分解：
├─ 发票抬头错误（含个人姓名、公司简称）：约 35%
├─ 票据日期超出出差范围：约 25%
├─ 费用类型与发票不匹配：约 20%
├─ 金额计算错误：约 12%
└─ 重复提交：约 8%
```

**关键洞察**：80% 的退回问题属于"规则明确、可自动判断"的合规问题，AI 具备完全替代人工初审的条件。

## 四、产品设计思路

### 设计原则

1. **AI 辅助，人做决策**：AI 给出建议和依据，最终决定权在财务专员——建立信任，降低推行阻力
2. **源头拦截优于末端审核**：在员工提交前就告知问题，而非等到财务退回后再修改
3. **每条结论可追溯**：所有 AI 判断都引用政策库原文，消除"AI 瞎说"的质疑
4. **渐进式自动化**：一期建立信任 → 二期批量放行绿单 → 三期全自动

### 用户分层与核心诉求

| 用户角色 | 表面需求 | 真实诉求 | 设计响应 |
|---|---|---|---|
| **员工**（报销提交者） | 快点打款 | 不要被退回重填 | 提交前 AI 预检，OCR 减少手工录入错误 |
| **财务专员** | 少审核 | 不背锅、有依据 | AI 给结论+政策原文，财务点确认而非从零判断 |
| **财务主管/CFO** | 合规达标 | 内审时说得清楚 | 每条审核留完整操作日志，异常率可视化 |

## 五、AI 能力赋能点设计

### 赋能点 ① — OCR 票据识别（员工侧）

**解决什么问题**：员工手工录入票据信息，错误率高，是退回率 20% 的源头之一

**AI 能力**：视觉大模型（豆包 Vision）读取发票图片，提取：
- 发票类型（增值税专票/普票/定额）
- 发票抬头（购买方名称）
- 金额（自动填充，避免手工输入误差）
- 开票日期
- 销售方名称 & 发票号码

**降级策略**：视觉模型不可用时，OCR 结果标注为"需人工确认"，不阻断提交流程

**边界条件**：
- 手写金额发票 → OCR 置信度低时强制人工复核
- 图片质量差（模糊/逆光）→ 提示用户重新拍摄
- PDF 发票 → 解析文字层，不走视觉模型

---

### 赋能点 ② — 提交前 AI 合规预检（员工侧）

**解决什么问题**：用户不熟悉报销政策，提交了才知道有问题

**AI 能力**：LLM 基于规则库，在提交前检查：
1. 发票抬头是否为公司名称
2. 票据日期是否在出差日期范围内（允许 ±1 天弹性）
3. 费用类型与发票税目是否匹配（餐饮费不可用办公用品发票）
4. 金额是否超过各类目政策上限
5. 是否存在重复发票（系统自动比对发票号）

**预期效果**：将退回率从 20% 降至 8% 以下（源头拦截约 60% 的问题）

---

### 赋能点 ③ — RAG 政策库检索（财务侧）

**解决什么问题**：AI 的判断缺乏可信度，财务无法核实 AI 依据

**AI 能力**：RAG（检索增强生成）

```
政策 Excel 文件
  ├─ 差旅标准.xlsx（住宿/餐饮/交通/出差天数）
  └─ 发票合规规则.xlsx（抬头/类型/特殊情形/金额规则）
          ↓
  TF-IDF 向量化（字符级 n-gram，无需模型下载）
          ↓
  审核时按报销内容检索相关条款
          ↓
  每条 AI 判断附带政策来源：
  「发票合规规则.xlsx · 发票抬头规则 · 第4行」
```

**设计亮点**：
- 政策库 Excel 可由财务主管直接维护，无需工程师介入
- 支持热更新：上传新 Excel → 调用 `/policies/reload` 接口即时生效
- 来源精确到"文件名·Sheet名·行号"，可对照原文核实

---

### 赋能点 ④ — 结构化风险评级（财务侧）

**解决什么问题**：财务面对大量"待审核"单据，不知从哪里开始

**AI 能力**：对每张报销单生成风险等级：

| 等级 | 触发条件 | 推荐处理 |
|---|---|---|
| 🔴 高风险 | 硬规则未通过（个人抬头、重复票、超差标五星酒店） | 人工重点审核 |
| 🟡 中风险 | 软规则异常（金额超限、日期边缘、类型不匹配） | 快速确认 |
| 🟢 低风险 | 所有规则通过 | 可设置批量放行（二期功能） |

**队列优先排序**：高风险 → 中风险 → 低风险，确保财务精力用在刀刃上

---

### 赋能点 ⑤ — AI 起草退回话术（财务侧）

**解决什么问题**：财务专员写退回通知措辞不一，有时过于生硬，员工体验差；有时过于模糊，员工不知道如何修改

**AI 能力**：LLM 根据具体问题，起草标准化退回话术：
- 承认员工辛苦，语气友善
- 指出具体问题（引用字段名和数值）
- 告知需要补充/修改的内容
- 100-150 字，一键复制发送

**风险控制**：话术是建议，不是自动发送；财务可以修改后再用

---

### 赋能点 ⑥ — AI 管理层月度分析（管理侧）

**解决什么问题**：财务主管每月手工汇总数据写报告，费时且缺乏洞察

**AI 能力**：基于月度聚合数据，生成：
- 整体表现评价（与目标对比）
- 异常率分公司拆解（识别哪个城市问题最多）
- Top 异常类型趋势分析
- 下月预测与改善建议

## 六、产品功能详解

### 功能地图

```
Expense-Agent
├── 员工提交端
│   ├─ 基本信息录入（姓名/部门/城市/出差日期）
│   ├─ 发票图片上传（OCR自动识别 → 一键填充）
│   ├─ 票据明细管理（增删改，合计自动计算）
│   ├─ AI 预检（提交前合规扫描，流式输出）
│   └─ 提交报销单（含图片附件ID）
│
├── 财务审核端
│   ├─ 报销单队列（高风险优先排序）
│   └─ 单据详情视图（双栏设计）
│       ├─ 左栏：报销信息 + 费用明细 + 发票原件
│       └─ 右栏：AI 预审结论
│           ├─ 结构化问题卡片（标题/风险/描述/政策来源）
│           ├─ AI 起草退回话术（流式生成）
│           └─ 操作按钮（复制话术并退回 / 特批通过）
│
└── 管理仪表盘
    ├─ 核心指标（总单数/总金额/审核时长/异常率）
    ├─ 分公司对比（单数/金额/异常率）
    ├─ Top 异常类型条形图
    ├─ 审核结果分布（通过/退回/待审）
    └─ AI 月度管理层摘要（一键生成）
```

### 关键用户旅程

**员工报销旅程**：
```
拍发票照片
  → OCR 识别（2秒）→ 自动填充字段
  → 点击「AI预检」→ 流式输出风险等级和具体问题
  → 按提示修改 → 提交
  → 收到进度通知（待审核/已通过/已退回+原因）
```

**财务审核旅程**：
```
登录，看到待审核队列（红单排前面）
  → 点击高风险单据
  → 左栏：查看报销详情，核对原始发票图片
  → 右栏：看 AI 问题卡片（附政策依据原文）
  → AI 话术已准备好
  → 一键「复制话术并退回」或「特批通过」
  → 回到列表，处理下一单
```

## 七、边界条件与风险控制

### AI 准确性风险

| 风险 | 场景 | 控制措施 |
|---|---|---|
| 误判合规为违规 | AI 将有效发票标记为问题 | 定位为"建议"而非"自动拒绝"；财务可一键特批通过 |
| 漏判违规为合规 | 高风险问题被评为低风险 | 硬规则（抬头错误/重复票）用确定性规则引擎，不走 LLM |
| LLM 幻觉 | 模型编造不存在的政策条款 | RAG 约束：AI 输出必须引用检索到的原文，无来源不输出结论 |
| 政策更新未同步 | 新差旅标准未更新到政策库 | 政策库支持热更新；在 AI 输出末尾标注"政策来源更新日期" |

### 数据安全风险

| 风险 | 控制措施 |
|---|---|
| 发票含员工个人信息（姓名/金额） | 私有化部署 LLM，数据不出企业网络 |
| 发票含供应商商业信息 | 同上；不使用公共 LLM API 处理原始票据 |
| 图片存储合规 | 服务器加密存储，设置 90 天自动清除 |

### 推行阻力风险

| 风险 | 场景 | 应对策略 |
|---|---|---|
| 财务专员抵制 | 担心被 AI 替代 | 强调"AI 给建议，人做决策"；AI 审核减少的是重复操作，不是财务职位 |
| 员工不信任 AI 预检结果 | 认为 AI 瞎说 | 每条问题附政策原文和来源；增加"忽略此提示"选项，不强制 |
| 各地政策差异 | 北京和成都差旅标准不同 | 政策库支持按城市分 Sheet 配置；RAG 检索时自动匹配城市 |

### OCR 精度风险

| 场景 | 处理方式 |
|---|---|
| 手写金额 | 标注为"需人工确认"，高亮显示 |
| 图片模糊/逆光 | 检测图片质量分，低于阈值时提示重拍 |
| 非标准发票（定额/手写收据） | 归类为"无法自动识别"，保留人工录入通道 |
| PDF 电子发票 | 优先解析文字层，失败再走视觉模型 |

## 八、成功指标

### 核心 KPI

| 指标 | 当前值 | 目标值（6个月） | 测量方式 |
|---|---|---|---|
| 财务人均审核时长 | 10-15 分钟/单 | < 5 分钟/单（-60%） | 系统打点记录 |
| 报销退回率 | 20% | < 8%（-60%） | 退回单数/总单数 |
| 月底积压率 | 月底集中爆发 | 全月均匀分布 | 每日审核单数分布 |

### 护栏指标（不可恶化）

| 指标 | 上限 | 说明 |
|---|---|---|
| AI 误判率（误拒合规单） | < 2% | 超过则关闭自动退回功能，降级为"建议" |
| 财务合规事故率 | 0 增长 | AI 漏判不应导致不合规票据通过 |
| 员工投诉率 | < 5% | 月度 NPS 调研 |

### 次要指标

- OCR 自动填充率（目标 > 80%）
- AI 话术使用率（目标 > 60%，即财务接受 AI 起草的话术）
- 政策库覆盖率（目标 100% 常见报销场景）

## 九、技术架构

### 整体架构

```
┌─────────────────────────────────────────────────┐
│                   前端（单页应用）                  │
│   员工提交端 │ 财务审核端 │ 管理仪表盘              │
│   Tailwind CSS + 原生 JS + marked.js             │
└────────────────────┬────────────────────────────┘
                     │ HTTP / SSE
┌────────────────────▼────────────────────────────┐
│               后端（FastAPI）                     │
│                                                  │
│  /upload/invoice  ← OCR 接口                     │
│  /submissions/*   ← 报销单 CRUD                  │
│  /policies/*      ← RAG 政策管理                  │
│  /dashboard/*     ← 仪表盘数据                    │
└──────┬─────────────┬───────────────┬────────────┘
       │             │               │
┌──────▼──────┐ ┌────▼─────┐ ┌──────▼──────┐
│  豆包 LLM   │ │ 视觉模型  │ │  RAG 引擎   │
│  (文本生成)  │ │  (OCR)   │ │ TF-IDF检索  │
│  流式输出   │ │ Base64图  │ │ Excel政策库 │
└─────────────┘ └──────────┘ └─────────────┘
```

### 关键技术决策

| 决策 | 选型 | 理由 |
|---|---|---|
| LLM | 豆包大模型（字节 ARK API） | OpenAI 兼容接口，国内网络稳定，数据不出境 |
| 流式输出 | SSE（Server-Sent Events） | 比 WebSocket 更轻量，实时展示 AI 输出，体验好 |
| 向量检索 | TF-IDF（scikit-learn） | 无需下载模型，启动快；政策文档结构化强，关键词匹配效果好 |
| 存储 | 内存（演示） / PostgreSQL（生产） | 演示用内存，生产环境切换成本低 |
| 政策库格式 | Excel | 财务团队直接维护，无需技术能力 |

### 核心数据流：AI 审核

```
用户点击「AI审核」
        ↓
1. RAG 检索：retrieve_for_submission(s)
   → 按费用类型+城市+已知问题 构建查询
   → 返回 Top-6 相关政策条款

2. 生成结构化问题卡片（不走 LLM，纯规则）
   → classify_issue(text) → (标题, 风险等级)
   → 为每个问题匹配 RAG 来源

3. SSE 推送给前端：
   event 1: {"structured_issues": [...]}  → 渲染问题卡片（即时）
   event 2-N: {"content": "..."}         → 流式渲染退回话术
   event DONE: "[DONE]"
```

## 十二、产品亮点总结（面试 Pitch 版）

**一个核心判断**：报销审核的痛点不是"人不够聪明"，而是"规则太多、票据太杂、人做重复判断太累"。AI 最适合解决的正是这类"规则明确但执行量大"的问题。

**三个设计原则**：
1. **源头拦截**：80% 的问题在员工提交前就能被 OCR + 预检发现，比末端退回成本低 10 倍
2. **来源可信**：每条 AI 判断附政策原文，财务不需要"相信 AI"，只需要"核实依据"
3. **渐进自动化**：先从"AI 建议、人决策"开始建立信任，再逐步开放自动放行

**三个 AI 核心能力**：
- **OCR**：把"拍票据"变成"系统自动填表"，消除录入错误源头
- **RAG**：把分散的 Excel 政策库变成可检索的知识库，每个结论都有依据
- **LLM**：把"财务写退回理由"变成"财务确认 AI 起草的话术"，节省沟通成本

**量化价值**：
- 审核效率提升 60%（10-15 分钟/单 → 5 分钟/单以内）
- 退回率降低 60%（20% → 8%）
- 2000人企业，月均节省财务工时约 **500 小时**
